Studio → Astro 移行の是非
— 検証レポートと推奨アクション

cynthialy.co.jp（Studio.Design 製）を Astro 静的サイトへ移行する是非を、約2週間の実装検証の結果に基づいて整理。社内意思決定のための判断材料として残す。

作成2026-04-30 対象cynthialy.co.jp リニューアル検討メンバー 検証期間2026-04-24 〜 2026-04-30

TL;DR — 結論サマリ

Studio で構築されたサイトを Astro へミラー方式で移植する技術的検証は完了。事例記事の量産フォーマット や ピクセル一致のミラー表示 は実現できた。一方で、ミラー HTML を部分的に編集しようとするとデザインが連鎖的に崩壊し、新規ページを Studio トーンと完全一致させることも難しい。

したがって移行を進めるなら、AI が安全に更新できるデザインシステム を Design.md として明文化し、コンポーネント方式で再構築する道しか現実解はない。Studio の見た目をそのまま完全保存しつつ AI で運用するハイブリッドは成立しない。

SECTION 01背景と検証の目的

cynthialy.co.jp は Studio.Design（ノーコード Web ビルダー）で構築・運用されている。 リニューアル検討では下記の課題が確認された。

  • パフォーマンス：JS バンドルが重く、LCP が遅い
  • 運用フォーマットの不統一：事例記事の構造がページごとに違い、量産しづらい
  • 編集効率の限界：Studio 上での画面操作が必要で、1 件あたりの工数が大きい
  • ロックイン：Studio のエコシステムに依存し、独自要件への拡張が困難

そこで「Studio から Astro 静的サイトへ移行できるか」を検証するため、リポジトリ kamechi0702/cynthialy-lp 上で実装検証を実施した。

SECTION 03検証結果（OK / NG マトリクス）

3.1 できたこと（OK）

ミラーサイトの完全構築（ピクセル一致）
OK

Studio が出力した SSR HTML をそのまま Astro の静的ページに取り込み、CSS バンドルも合わせて配置することで、本家と ピクセルレベルで同じ見た目を再現できた。

成果物：TOP / サービス一覧 / コンサル / 研修 / 事例 × 2、計 7 ページが本家と同等の見た目で表示。背景動画・ロゴカルーセル・スクロール連動フェードインも動作。
事例記事の量産フォーマット（MDX + テンプレート）
OK

ミラーテンプレートに {`{{COMPANY}}`} {`{{TOC_*}}`} 等のトークンを埋め込み、frontmatter から流し込む方式が確立。MDX で本文だけ書けば、Hero / 目次 / 関連事例 / フッターが自動的に統一された見た目で生成される。

運用効率：1 件の事例記事を ~30 分で公開可能（従来の Studio 編集と比較して 1/3 の時間）。AI による下書き生成も MDX 形式なら容易。
インフラ・運用基盤の整備
OK

GitHub Actions による自動デプロイ（push → 約 50 秒で本番反映）、Design.md による設計憲章、AI 画像生成スクリプト（OpenAI gpt-image-1）の汎用基盤、 までを 1 セッション内で構築完了。

運用フロー：MDX を書いて git push すれば本番反映。新ページ用の写真は npm run generate:images でブランドトーン統一の AI 画像が出力。

3.2 できなかったこと（NG）

ミラーで構築したサイトの編集（部分修正でデザイン崩れ）
NG

ミラー HTML を「内容だけ差し替える」試み（わざツグ初版で実施）は、テキストは入れ替わるがデザインの語り口が Studio のままで、新コンテンツの訴求が立ち上がらなかった。さらに細部編集を進めるとレイアウトが崩れていく。

原因：Studio CSS は data-r-XX_YY_..._UUID という位置依存のセレクタを使用。要素を移動・追加・削除すると階層パスが変わり、対応する CSS がマッチしなくなる。AI が部分的にテキストを差し替えるだけでも、全体のレイアウトが連鎖的に崩れるリスクがある。
事例ページの CTA / フッターの統一
NG

事例ページのフッターはメインページと異なり、CTA セクション・Bottom nav が欠落していた。ミラー HTML から該当部分を抽出して移植する試みは、CSS 依存（.container の幅制約・data-s ベースのレイアウト 463 ルール・media query 287 個）が複雑すぎて、移植先で必ず崩れた。

最終的な解決：ミラー方式を諦め、Design.md 準拠で Footer.astro を新規実装。デザイントーンは揃えたが、Studio との完全一致は犠牲になった。

3.3 部分的にできたこと（Partial）

新規プロダクトページ（わざツグ）の生成
PARTIAL

cynthialy.co.jp に存在しない新規プロダクト「わざツグ」のページを Design.md のセクションアーキタイプに沿ってコンポーネント方式で実装。深緑×ゴールドのブランドトーンは保ったが、Studio タイポグラフィや細部の余白などは完全には合わせきれなかった。

制約：Studio の独自フォント（秀英角ゴシック銀・FontPlus 契約フォント）はライセンスの都合で再現不可。Design.md で代替（Noto Sans JP + Inter）を定義したが、印象は微妙に変わる。

SECTION 04技術的に明らかになったこと

  • Studio の CSS は位置依存（data-r-...）と要素 ID 依存（data-s-...）の二重構造で、合計 700+ 個の inline <style> ルールが各ページに散らばっている
  • これらのルールは親 div の階層パスにロックされているため、要素を切り出して別ページに移植すると親階層が変わり、CSS が無効になる
  • ミラーで取り込んだ HTML を AI で書き換えるためには、対応する CSS ルールも全て検出して同時に書き換える必要があり、編集の一貫性を保つ自動化は事実上不可能
  • 一方、Design.md ベースのコンポーネント実装ならCSS の cascade が予測可能で、AI が安全に部分編集できる
  • Studio タイポグラフィや独自フォントの完全再現は、ライセンス・モバイル対応の両面で不可

SECTION 053 つの選択肢

PATH A
Studio 現状維持
（人がノーコードで運用）
初期コスト：0 ／ 移行リスク：なし
  • cynthialy.co.jp をそのまま使い続ける
  • 編集はメンバーが Studio 上で実施
  • 運用効率・パフォーマンス課題は据え置き
PATH B
Astro 移行 ＋ 人で運用
初期コスト：高 ／ 月次運用：中
  • Astro で再構築するが、編集はエンジニアが手動
  • パフォーマンスは改善
  • 運用工数は Studio 時代と変わらない or 増える
PATH C
Astro 移行 ＋ AI で運用
（Design.md 駆動）
初期コスト：高 ／ 月次運用：低
  • Design.md と Astro コンポーネントで再設計
  • 記事追加・改修は AI が安全に実行可能
  • 運用効率・パフォーマンス・拡張性すべて改善
  • 移行するならこれ以外の道はない

SECTION 07AI 運用 vs 人運用 メリット・デメリット

7.1 AI による運用（Path C）のメリット・デメリット

MERITS

  • 記事生成・改修の速度が圧倒的（30 分／本）
  • Design.md 準拠で一貫性が機械的に保たれる
  • 24/365 稼働、夜間・休日のリリースも可
  • 差分は git 履歴に残るため監査・ロールバック容易
  • 新ページ・新機能の追加コストが下がる
  • 画像生成・コピー検討も同じワークフローで完結

DEMERITS

  • 初期に Design.md とコンポーネント基盤の整備が必要（中規模工数）
  • AI に指示する人の側に「ブランド理解」と「指示力」が要る
  • 完全な自由デザインからDesign.md の枠内に収まる制約を受け入れる必要あり
  • AI 生成物のレビュープロセスが必要（誤情報や事実関係のチェック）
  • OpenAI など外部 API 利用料（軽微：月数千円〜数万円）

7.2 人による運用（Path A / Path B）のメリット・デメリット

MERITS

  • ノーコード（Studio）なら非エンジニアでも触れる
  • 細部のクリエイティブ表現の自由度が高い（手作業ゆえ）
  • 新規メンバーのオンボーディングが容易（Studio の場合）
  • 外部依存（AI API）が無く運用コストが読める

DEMERITS

  • 記事 1 本あたりの工数が大きい（1〜2 時間）
  • 担当者によって品質・スタイルがブレる
  • Astro 移行（Path B）の場合、エンジニア人件費が継続発生
  • 属人化のリスク（特定メンバー退職時の引き継ぎ難）
  • Studio 維持（Path A）の場合、性能・運用効率の課題が解決しない
  • 月次の更新頻度が高くなると頭打ちになりやすい

SECTION 08結論と推奨アクション

検証結果を踏まえると、選択肢は実質的に Path A（現状維持） か Path C（Astro + AI 運用） の二択である。Path B（Astro + 人運用）は初期コストだけ Path C と同等にかかり、運用コストは Path A 並みに高止まりするため、合理的な選択肢ではない。

そして、Studio の見た目を完全に保ったまま AI で更新するハイブリッド運用は、本検証で技術的に成立しないことが明確になった。CSS の位置依存セレクタによって、AI の小さな編集すら全体崩壊を引き起こす。

RECOMMENDATION

移行するならば、AI が安全に更新できるデザインで構築すること が大前提。すなわち Path C（Astro + Design.md + AI 運用）を採る。

移行する/しないの判断は 「月次の更新頻度・新規ページ追加頻度・運用効率の重要度」を基準にすべきである：

  • 月 5 件以上の更新があり、新ページ拡張を継続的に行う → Path C を推奨
  • 月 1〜2 件で十分、当面は現体制で回せる → Path A で延命も合理的
